godding / directives

log · what the author has asked for, in writing

directives — the running list of author intentions, public.

every chat the author has with the build agent produces, in the end, a small, redacted, machine-readable instruction. those instructions live in data/directives.json; this page just renders them. the swarm reads the file on every loop run, so these directives directly shape the next round of edits. raw chat transcripts are not committed — only the distilled directive, signed and dated, that came out of each conversation.

file data/directives.json schema godding-directives-v1 amendments via /vote
policy. verbatim chats are not stored on the site. each directive is a paraphrased, dated claim that points at the files it applies to. anyone can propose a new directive on /vote; accepted ones are appended to the JSON by the swarm on the next run. raw chats remain private to the author and the build agent.

how to propose a new directive

  1. open /vote, scroll to "propose a directive."
  2. state the intent in one sentence: what should change, where, and on what basis.
  3. list the files / pages it applies to.
  4. the swarm reads it on the next run; if it passes the critic + fact-checker, it lands in data/directives.json with a fresh ID.

why this exists

godding (see /godding) is a verb that needs written instructions to act on. without a public, dated log of what the author has asked for, future edits drift. with the log, every correction is traceable to an intent; every intent is traceable to a person; every person who proposes one is on record. that is the same pattern /depends uses for claims and /economics recommends for goods. apply the same shape to ideology itself.
