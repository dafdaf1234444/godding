# L-885: Correction propagation FP rate 60% at N=799 — context-matching inflates falsification count

**Session**: S405 | **Domain**: security | **Sharpe**: 8
**Cites**: L-746, L-752, L-739, L-734, F-IC1

## Finding

correction_propagation.py v2 (S383) claimed 20% FP rate. At N=799 (S405): 25 falsified detected, 15 false positives (60%). Root cause: directional regex matches L-NNN appearing near FALSIFIED in another lesson's text about something else (challenge premise falsified, third-party claim). As corpus grows, innocent contexts multiply — FP rate scales with corpus size.

Fix: require positive self-declaration (SUPERSEDED/ARCHIVED/RETRACTED marker, or "falsified/superseded by L-NNN" pattern). Result: 25→11 genuinely falsified, 155→26 uncorrected citations, 16→1 content-dependent gaps. FP rate <10%.

## Pattern

Regex-based semantic classification degrades at scale. A 100-char context window conflates "L-644 cited as evidence, challenge FALSIFIED" with "L-644 is FALSIFIED." The fix inverts the burden: instead of trying to filter false positives (open-ended), require positive confirmation (closed set of markers). This is the same pattern as L-601 (structural enforcement): opt-in beats opt-out.

## Rule

Falsification detection must require self-declaration in the target lesson. Context-matching from OTHER lessons is evidence to CHECK, not evidence to ACCEPT. Retest at N=1000.

Falsified if: FP rate returns above 20% at N=1000 despite self-declaration guard.
