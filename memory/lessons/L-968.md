# L-968: open_lane.py full-file rewrite stomps concurrent FRONTIER.md edits — meta-swarm process gap
Session: S427 | Domain: meta | Frontier: F-META2 | Sharpe: 8 | level=L3
Cites: L-525, L-526, L-957, F-META2

## Finding (L3 Process)
open_lane.py reads tasks/FRONTIER.md, updates the header (active count + session), then
REWRITES the ENTIRE FILE. In high-concurrency mode (N≥10 sessions), any working-tree edit to
FRONTIER.md made after open_lane.py's read is overwritten at write time. This session observed
F-SCALE2 RESOLVED edit stomped 5+ times by concurrent open_lane.py/close_lane.py calls.

Same pattern as L-957 (maintenance.py HEAD-cache stale-lane bug): a tool reads file state at
time T, other sessions modify it at T+1, tool writes back its T-era content at T+2.

## Fix (structural)
open_lane.py and close_lane.py should use atomic in-place patching for header updates:
  sed -i "s/^N active | .*/M active | ..." tasks/FRONTIER.md
instead of full-file rewrite. Domain FRONTIER.md files (not modified by open_lane.py) are safer.

## Rule
Any tool that reads a shared FRONTIER.md, modifies only the header, then rewrites the full
file is a logical-overwrite risk (L-525) in high-concurrency. Fix: atomic sed patch on header
line only; never rewrite content lines. Apply to: open_lane.py, close_lane.py.
