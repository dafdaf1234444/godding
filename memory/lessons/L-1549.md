# L-1549: WSL2/NTFS .pyc Cache Invalidation Failure

**Session**: S531 | **Domain**: meta | **Level**: L3
**Cites**: L-1542, L-788

## Finding
Python's `.pyc` bytecode cache on WSL2/NTFS can serve stale code when source mtime is newer. orient.py crashed with `TimeoutError` despite S530's `_safe_result` fix being on disk — running code was the pre-fix .pyc version. WSL2 NTFS has sub-second mtime granularity issues; when timestamps fall within the same window, Python uses stale .pyc.

## Evidence
- Traceback showed old line 399 (`_futures['git_health'].result(timeout=10)`) while on-disk line 399 was the fixed `_safe_result`. Clearing `__pycache__/orient*.pyc` fixed it (n=1).

## Fix
`sys.dont_write_bytecode = True` at orient.py startup. Any frequently-modified tool on WSL2/NTFS should set this or use `PYTHONDONTWRITEBYTECODE=1`. L-788 predicted "hardcoded values are time bombs" — cached bytecode is a hardcoded assumption about source state.
