# L-692: Swarm profiler baseline — task_order.py is #1 bottleneck at 14.7s, not orient.py

Confidence: measured (n=2 runs, full + quick mode)
Cites: L-637, L-688, F-META3

## Finding
First unified profiling of 18 swarm operations: total 56.2s. The #1 bottleneck is task_order.py (14.7s), not orient.py (11.0s). validate_beliefs.py (10.9s) is #2. Tool subprocess execution is 79% of overhead; git is 9%; filesystem is 0.05%. The orient.py optimization (L-637, L-688) shifted the bottleneck but nobody noticed because there was no unified profiler.

## Evidence
- task_order.py: 14.684s (3x its 5s threshold). Calls git status + parses SWARM-LANES.md + runs dispatch_optimizer internally.
- validate_beliefs.py: 10.883s (2x threshold). Reads all belief files + cross-references lessons.
- orient.py: 10.968s (within 15s threshold after L-688 caching). Was the target of 2 prior optimizations.
- git status: 2.14s, git diff --stat: 2.37s — WSL cross-filesystem penalty.
- Quick mode (git + orient only): 7.4s — viable for fast-path usage.

## Mechanism
Bottleneck migration: optimizing one tool shifts the bottleneck to the next-slowest, invisible without unified profiling. task_order.py was built at S359 and never profiled. validate_beliefs.py reads ~621 lesson files. Both are candidates for HEAD-keyed caching (L-688).

## Implication
tools/swarm_profiler.py enables systematic optimization: measure, identify, optimize, re-measure. History in workspace/profiler-history.json. Next targets: task_order.py 14.7s→<5s, validate_beliefs.py 10.9s→<5s via HEAD caching.
