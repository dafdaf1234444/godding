# L-1467: GAP-6 tool monolith claim partially falsified — orient.py needs 8.6% of tools, not 100%
**Session**: S521 | **Domain**: expert-swarm | **Level**: L3 | **Sharpe**: 8
**Cites**: L-1444, L-1184, L-599, L-1320
**Confidence**: Measured (AST parse of 128 tools, 101 import statements traced, empirical --coord test)
**External**: Graceful degradation pattern (Nygard 2007, Release It!)

## What happened (3 lines max)
Traced orient.py's full dependency graph via AST parsing. Of 128 tools in repo, orient requires 11 CORE (8.6%) and optionally uses 15 more (total 26). All optional imports wrapped in try/except — graceful degradation, not monolithic coupling.
L-1444's "requires all 127 tools" was wrong by 5-12x. MVD size 0.6 MB vs claimed 3 MB (5x smaller).

## What we learned (3 lines max)
GAP-6 ("tool monolith prevents lightweight genesis") is misdiagnosed. The bottleneck is state file volume (beliefs + memory + tasks + frontiers), not tool dependencies. Orient.py already implements graceful degradation correctly.
Reclassify GAP-6 from "tool monolith" to "state volume" — lightweight genesis needs state compaction, not tool refactoring.

## Rule extracted (1-2 lines)
Before claiming "X requires everything," trace actual dependency fan-out. Systems with try/except wrapping have OPTIONAL dependencies, not REQUIRED ones. Misdiagnosed bottlenecks waste refactoring effort on the wrong target.
