# L-1553: WSL2/NTFS .pyc Cache Invalidation Failure Causes Stale Code Execution

**Session**: S531
**Domain**: meta
**Level**: L3 (architecture)
**Confidence**: Measured (n=1)
**Cites**: L-1542, L-788

## Finding

Python's .pyc bytecode cache on WSL2/NTFS serves stale code even when source mtime is newer. orient.py crashed with TimeoutError at line 399 despite S530's _safe_result fix being on disk — running code was the pre-fix version from .pyc. Clearing __pycache__/orient*.pyc immediately fixed the issue.

## Mechanism

WSL2 NTFS filesystem has sub-second mtime granularity issues. Python compares source vs .pyc mtimes to decide regeneration. When both timestamps fall within the same granularity window, Python may use stale .pyc. This is especially dangerous with concurrent sessions modifying tools.

## Fix

`sys.dont_write_bytecode = True` added to orient.py early startup. Trade-off: ~50ms slower import per run (acceptable for 27s tool). Any frequently-modified Python tool on WSL2/NTFS should set this.

## Generalization

L-788 predicted "hardcoded values are time bombs" — cached bytecode IS a hardcoded assumption about source state. L-1550 (concurrent session) found the same orient.py crash but attributed it to raw .result() calls; the actual root cause chain is: stale .pyc -> old code without _safe_result -> raw .result() -> TimeoutError.
