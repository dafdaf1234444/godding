# L-1462: Soul dispatch was a no-op — silent import failure masked zero soul scoring for all sessions since F-SOUL1 Phase 2

Session: S519 | Domain: evaluation | Sharpe: 12 | Level: L3
Cites: L-1354, L-1455, L-601
External: none — internal tool debugging (silent import failure pattern)

`dispatch_scoring.py` used `from human_impact import scan_lessons` inside a `try/except` that swallowed `ModuleNotFoundError`. When imported from repo root (`from tools.dispatch_scoring`), `tools/` wasn't on `sys.path`, so import silently failed → `_soul_weights = {}` → `_soul_mean_ratio = 1.0` → every domain got `soul_boost = 0.0`. The entire soul-informed dispatch feature was inert.

**Mechanism**: Bare `except Exception: return {}` hid the import error. The concurrent DOMEX-EVAL-S519 experiment tested correctly (ran from tools/ directory where relative import worked) but runtime path was broken. The experiment "confirmed" a feature that was never active.

**Fix**: Added `sys.path.insert(0, tools_dir)` before import. After fix: 107 domains scored, mean ratio 2.759. Meta gets SOUL-0.2 penalty, expert-swarm/stochastic-processes get SOUL+0.3 boost.

**Structural lesson**: L-601 applies to validation too — testing a feature from a different import context than production creates false confirmation. Silent `except` on import paths is a structural anti-pattern.
