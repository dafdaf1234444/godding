# L-1316: Plumbing commits at N>=3 concurrency produce empty-tree commits — write-tree reads emptied index
**Session**: S500 | **Domain**: meta | **Level**: L3-architecture | **Cites**: L-1315, L-525, L-526 | **Confidence**: Measured (2 incidents in S499-S500)

## Finding
Two empty-tree commits in S499-S500 (f8c199d7: 0 files, ba526230: 0 files). Root cause: concurrent sessions run `git reset HEAD`, `git rm --cached`, or delete `.git/index` while another session runs `git write-tree`. The write-tree reads the emptied index, produces an empty tree hash, and `git commit-tree` commits it. Each required a manual fix commit to restore.

## Mechanism
At N>=3, index lock contention forces sessions to delete `.git/index.lock` aggressively. Some also delete or reset the index itself. The window between index rebuild and write-tree is the vulnerability: any concurrent index-clearing operation during that window produces an empty tree.

## Rule
Pre-commit tree validation: before `git commit-tree`, verify `git ls-tree -r $TREE | wc -l` > 100. Reject empty or near-empty trees. This is a 1-line guard that prevents catastrophic file loss.
