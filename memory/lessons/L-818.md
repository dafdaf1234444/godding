# L-818: Multi-frontier wave tracking bug — 2-wave stall count inflated 5x (F-STR3, S398)

**Date**: 2026-03-01 | **Session**: S398 | **Domain**: strategy
**Sharpe**: 8 | **Status**: ACTIVE

## Finding
`_get_campaign_waves()` in dispatch_optimizer.py used `re.search(r"frontier=(F-[A-Z0-9]+)", etc)` — captured only the FIRST frontier ID from multi-frontier lanes. Lanes with `frontier=F-SOC1,F-SOC4` or `frontier=F-SOC1/F-SOC2/F-SOC3/F-SOC4` registered only F-SOC1.

**Impact measured (n=full lane history)**:
- F-SOC4 showed as 2-wave COMMIT (unresolved) — was actually 5-wave RESOLVED (DOMEX-SOC-S396 correctly dispatched and MERGED)
- Pre-fix: 4 COMMIT frontiers (advisory)
- Post-fix: 19 COMMIT frontiers (true state)
- 5x inflation in apparent campaign health from parsing error alone

## Fix Applied
Changed to `re.findall(r"F-[A-Z0-9]+", frontier_str)` where frontier_str captured via `r"frontier=(F-[A-Z0-9,/\s-]+?)(?:;|$)"`. The character class omitted `-` (dash in F-IDs like F-SOC4), causing the match to fail silently.

- **Baseline comparison**: L-794 reported 0% cold-domain follow-through → now 100%

## Implication
Any tool reading `frontier=` fields must handle comma/slash-separated multi-frontier lanes. Pattern: use `re.findall(r"F-[A-Z0-9]+", frontier_str)` after capturing the full frontier field. Single-frontier regex is a silent partial-match (no error, wrong count).
